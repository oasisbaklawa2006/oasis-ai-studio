## 5) Current Supabase setup
- Frontend uses `@supabase/supabase-js` client initialized from:
  - `VITE_SUPABASE_URL`
  - `VITE_SUPABASE_PUBLISHABLE_KEY`
- Client auth config persists session in `localStorage` with auto-refresh enabled.
- Supabase usage spans:
  - table CRUD in many pages/components,
  - RPC functions (`get_current_user_roles`, `generate_oasis_sku`, etc.),
  - storage bucket uploads (`product-media`),
  - at least one edge function (`supabase/functions/test-integration`).
- `supabase/config.toml` and multiple migrations exist in-repo; no new migration to be written in this task.

## 6) Current auth flow
- Auth state is provided globally by `AuthProvider`.
- Session restored via `supabase.auth.getSession()` on app boot.
- Role model loaded through RPC `get_current_user_roles`.
- Route gating:
  - `ProtectedRoute` requires authenticated session.
  - `RoleGate` checks page-level authorization by role.
- Sign-out is via `supabase.auth.signOut()`.
- Additional Lovable OAuth bridge exists (`src/integrations/lovable/index.ts`) and should be retired in rebuild.

## 9) Mobile overflow risk areas
Observed high-risk patterns:
- Dense table/grid-like editors with many inputs in one row:
  - `ChannelPricingRules`, `ChannelMoqRules`, `BomBuilder`, label and media editors.
- Long identifier strings/URLs (media URLs, SKUs, slugs) in cards/tables.
- Sidebar + content with many control clusters (possible width compression issues).
- Some protection already exists globally (`overflow-x: hidden`, `min-w-0`, responsive paddings), but component-level overflow needs targeted review.

## 10) Tab-refresh/state-reset risk areas
- Significant state is local component state without URL/search-param persistence.
- Multi-step edits may be lost on refresh in editors (Product edit, BOM, pricing/MOQ rule forms, media upload staging, label workflow).
- Auth/role loading can still cause temporary gating/visibility transitions at startup.
- React Query is configured but much of data fetching/mutation is still manual (stateful `useEffect` + direct calls), increasing reset/reload fragility.

## 11) Existing UI system/theme structure
- Tailwind + custom CSS variables theme in `src/index.css`.
- Brand tokens include warm ivory + teal + gold palette, gradients, custom shadows, and typography pairing (Cormorant Garamond + Inter).
- Component primitives are shadcn/Radix based (`src/components/ui/*`), with utility-based composition.
- Layout pattern: persistent sidebar + responsive mobile header shell.

## 12) Recommended clean architecture
Recommended target (incremental, no immediate behavior rewrite):
1. **Core layers**
   - `domain/`: pure types and business rules.
   - `application/`: use-cases/services (orchestrate reads/writes, validation, policies).
   - `infrastructure/`: Supabase adapters (queries, RPC, storage), auth adapters.
   - `presentation/`: pages/components/hooks.
2. **Data boundary**
   - Remove direct DB writes from UI components; route through typed service functions.
3. **State strategy**
   - Standardize server state in React Query; retain UI-only ephemeral state locally.
4. **Auth strategy**
   - Remove Lovable Cloud dependency; keep direct Supabase OAuth/session model.
5. **Observability**
   - Centralize error handling + toast mapping + audit instrumentation.

## 15) Recommended migration strategy for Central Supabase
No SQL authored here; strategy only:
1. **Inventory and map** existing tables/RPC/storage/RLS dependencies from current code.
2. **Define target canonical schema** for Central Supabase (namespaces, ownership, policies).
3. **Gap matrix**: current->target for every table, RPC, enum, policy, and storage bucket.
4. **Compatibility layer phase**:
   - maintain temporary views/functions where needed to avoid frontend breakage.
5. **Data migration phases**:
   - seed reference tables,
   - migrate master entities,
   - migrate relationship tables,
   - reconcile media paths and public/private access.
6. **Cutover controls**:
   - feature flags + read-only freeze window + rollback plan.
7. **Post-cutover validation**:
   - row counts, checksums/samples, auth/RLS smoke tests, route-level UAT.
